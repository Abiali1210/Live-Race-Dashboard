## Decisions Made

- Keep `/health` minimal for simple liveness checks.
- Use `/api/status` for detailed debugging so frontend/backend troubleshooting does not require inspecting the large `/api/state` payload.
- Keep websocket-specific diagnostics inside `wigeClient.ts` to avoid leaking socket lifecycle details across modules.
- Keep race-state summary inside `raceState.ts`, because it owns the in-memory state shape.
- Avoid adding a separate types file for status in this chunk; local exported types are sufficient and keep the change small.

## Gotchas / Risks

- `/api/status.wige.connected` is derived from the current websocket `readyState`; `/api/status.raceState.connected` comes from race state. They should normally match, but status is now the better place for operational debugging.
- WIGE sends control/unknown messages too; `other` may be non-zero and is not necessarily a problem.
- `lastError` is cleared on a successful open. If you need historical errors later, add an error count/history field.
- The live smoke test may leave short-lived `TIME_WAIT`, `CLOSE_WAIT`, or `FIN_WAIT_2` socket rows briefly; this does not mean the backend is still listening. Check specifically for `LISTEN` on port `3001` if needed.
- `stopWigeClient()` still is not wired into graceful process shutdown.
