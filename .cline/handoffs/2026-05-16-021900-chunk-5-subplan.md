## Decision Made: Split Chunk 5 Into Three Subchunks

The original Chunk 5 is important and touches a real external API plus nested data normalization, so it is now split into:

1. **Chunk 5.1 — Metadata module skeleton + API contract, no network yet**
2. **Chunk 5.2 — Defensive Eventhub fetch + graceful failure handling**
3. **Chunk 5.3 — Full Eventhub metadata extraction, merge, and coverage checks**

Rationale:

- Keeps each step small, understandable, and independently testable.
- Separates Express/API wiring from external network behavior.
- Separates external fetch reliability from the more complex nested Eventhub normalization.
- Reduces risk of introducing multiple bugs at once.

### Chunk 5.2 — Defensive Eventhub fetch + graceful failure handling

Goal:

- Add the real Eventhub bootstrap network request while keeping extraction minimal.

Files:

- Update `backend/src/eventhubMetadata.ts`.
- `backend/src/server.ts` should already have startup wiring from 5.1; adjust only if needed.

Implementation details:

- `loadEventhubMetadata()` should fetch `config.eventhubBootstrapUrl` once on startup.
- Check HTTP status. Non-2xx responses should set `error` and keep cars empty.
- Parse JSON defensively. Invalid JSON or unexpected top-level shape should set `error` and keep backend alive.
- Store a user-readable error string in memory if the fetch fails.
- On successful fetch, set `loadedAt` to an ISO timestamp.
- Keep normalization intentionally minimal in 5.2. Suggested options:
  - either verify that the expected `vertical` object exists without extracting cars yet, or
  - extract only the easiest car records from `vertical.entrylist` if the shape is clear.

### Chunk 5.3 — Full Eventhub metadata extraction, merge, and coverage checks

Goal:

- Complete the original Chunk 5 responsibilities by producing a useful `Record<string, CarMetadata>` from Eventhub bootstrap data.

Files:

- Update `backend/src/eventhubMetadata.ts`.
- Update `backend/src/server.ts` only if the endpoint shape needs a small adjustment.

Eventhub source locations to inspect and merge:

```text
vertical.entrylist
vertical.events[*].sessions[*].entry_list
vertical.events[*].sessions[*].results[*].entrant
vertical.drivers
```

Implementation details:

- Normalize all usable car records into `MetadataByCarNumber`, keyed by car number string.
- Each `CarMetadata` should match `backend/src/types.ts`:
  ```ts
  {
    carNumber: string;
    entrantId: string | null;
    teamName: string | null;
    teamColor: string | null;
    className: string | null;
    make: string | null;
    model: string | null;
    drivers: string[];
    carshotUrl: string | null;
    carshotUrlFull: string | null;
    headshotUrls: string[];
  }
  ```
- Merge records conservatively:
  - Prefer existing non-empty values over `null`/empty values.
  - Combine driver names without duplicates.
  - Combine headshot URLs without duplicates.
  - Never throw because one record is malformed.
- Count `withImages` when `carshotUrl` or `carshotUrlFull` is present.
- Keep endpoint response:
  ```json
  {
    "count": 289,
    "withImages": 161,
    "loadedAt": "ISO_DATE",
    "error": null,
    "cars": {
      "80": {
        "carNumber": "80",
        "entrantId": "177",
        "teamName": "Mercedes-AMG Team RAVENOL",
        "teamColor": null,
        "className": "SP 9",
        "make": "Mercedes-AMG GT3",
        "model": null,
        "drivers": ["Maro Engel", "Luca Stolz", "Fabian Schiller", "Maxime Martin"],
        "carshotUrl": null,
        "carshotUrlFull": "https://.../80.jpg",
        "headshotUrls": []
      }
    }
  }
  ```

Tests for 5.3:

1. `cd backend; npm run typecheck`
2. Temporarily start backend with `npm start`.
3. Request `http://localhost:3001/api/metadata`.
4. Confirm response has `count`, `withImages`, `loadedAt`, `error`, and `cars`.
5. Confirm sample cars such as `80`, `1`, and `911` exist if current Eventhub data still contains them.
6. Confirm car `80` has `carshotUrl` or `carshotUrlFull` if current data still provides it.
7. Confirm cars without images use explicit `null` image fields, not missing fields.
8. Confirm backend still starts gracefully if Eventhub fetch fails, for example by temporarily using an invalid URL or otherwise simulating failure.
9. Confirm no WIGE websocket code has been introduced.

Success criteria for 5.3:

- Metadata loads from Eventhub at startup.
- Endpoint returns useful metadata counts and a car map.
- Missing external data is normalized to `null` or `[]`.
- Backend starts even if Eventhub fetch fails.
- No live WIGE websocket logic is introduced.
